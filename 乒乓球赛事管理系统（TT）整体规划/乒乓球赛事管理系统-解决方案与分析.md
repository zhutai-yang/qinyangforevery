## 一、需求归纳

TT 的产品定位调整为：**明星选手内容运营后台 + 高级展示官网**。

| 能力域 | 新定位说明 |
|--------|------------|
| **明星选手运营** | 以重点选手为主资产，维护档案、头图、定位语、代表高光、商业预告与外部链接 |
| **内容运营** | 文章、资讯、外部动态、首页区块围绕选手与热点故事编排 |
| **高级展示官网** | 首页与球员页优先呈现人物、视觉资产、高光内容和商业动线，赛事/成绩作为可信背书 |
| **赛事记录底座** | 保留赛事、赛程、成绩、场馆数据，用于支撑选手履历、过往战绩、当前状态与赛事预告 |
| **外部数据补充** | 外部动态作为内容线索和资讯流，不直接覆盖本地权威成绩 |
| **站点划分** | **高级展示官网（公众端）** + **明星选手内容运营后台（运营端）** |

## 二、行业常见设计参考（通用做法）

实际在用的体育/赛事类站点，多采用以下组合，可对照本方案取舍。

### 2.1 双端分离：公众站 + 管理后台

| 端 | 典型职责 | 常见技术形态 |
|----|----------|--------------|
| **公众官网** | 浏览赛事、赛程、成绩、球员页、资讯；SEO、分享 | 静态加速 + SSR/模板 或 SPA + 预渲染 |
| **管理后台** | 录入/审核赛事、赛程、成绩、公告；配置抓取任务、关注球员 | 强鉴权、表单密集，多为 SPA 或传统后台模板 |

**结论**：与您「官网 + 管理后台」拆分一致，也是业界最普遍做法。

### 2.2 数据来源三类模型

| 模型 | 说明 | 适用场景 |
|------|------|----------|
| **纯人工录入** | 后台维护全部数据 | 自有赛事、无稳定外部接口 |
| **外部同步 + 人工修正** | 定时/手动拉取官网或开放 API，冲突时以规则或人工为准 | 希望展示「国际乒联/WTT/协会官网」最新动态 |
| **混合** | 核心赛程/成绩以录入为准，资讯/链接用抓取或 RSS 补充 | 平衡成本与时效性 |

**关于「从官网拉取」**：  
- 若目标站点**无官方开放 API**，常见做法是 **定时爬虫 + 解析 HTML**（需注意 **robots.txt、版权与服务条款、频率限制**）。  
- 若有 **RSS、公开日历、或合作数据接口**，应优先走 **合法、稳定** 的渠道。  
- 生产上普遍会加：**抓取任务表、失败重试、变更 diff、只读展示区与「本站录入」数据分区标识**，避免覆盖人工精心维护的主数据。

### 2.3 内容类：预告与资讯

| 类型 | 通用设计 |
|------|----------|
| **赛程预告** | 来自结构化数据：`sch_match` 未开赛记录 + 筛选条件（赛事、球员、时间范围） |
| **编辑类预告/头条** | 独立 **CMS 文章或公告表**（标题、摘要、封面、发布时间、置顶、关联赛事/球员） |

这样「最新赛事」既可来自 **抓取的外部列表**，也可来自 **本站公告**，后台统一编排展示顺序。

### 2.4 球员页（人物聚合）

通用模式：**球员主数据** + **多源关联**：

- 关联 **报名/场次参与方**（您库中 `reg_athlete`、`sch_match_participant`）→ 自动汇总其赛程与成绩。  
- 可选 **「关注球员」配置**（后台勾选重点球员）→ 官网首页/预告模块优先展示。  
- 外部抓取的新闻若带球员名，可通过 **标签或模糊匹配** 挂到球员页（进阶，一期可不做）。

## 三、推荐总体架构

```
                    ┌─────────────────────────────────────┐
                    │           管理后台（需登录）            │
                    │  赛事/赛程/成绩录入 · 公告 · 抓取配置    │
                    └──────────────────┬────────────────────┘
                                       │
                    ┌──────────────────▼────────────────────┐
                    │   应用服务（Spring Boot + /api）        │
                    │  业务 API · JWT 鉴权 · 抓取（可选）     │
                    └──────────────────┬────────────────────┘
                                       │
          ┌────────────────────────────┼────────────────────────────┐
          ▼                            ▼                            ▼
    ┌───────────┐              ┌───────────────┐              ┌─────────────┐
    │  MySQL    │              │ 外部数据源     │              │ RabbitMQ     │
    │ 核心+限流表 │            │ HTTP 抓取等    │              │ 异步同步可选  │
    └───────────┘              └───────────────┘              └─────────────┘
                                       │
                    ┌──────────────────▼────────────────────┐
                    │           公众官网（只读为主）            │
                    │  首页 · 赛事 · 赛程/成绩 · 球员 · 资讯   │
                    └────────────────────────────────────────┘
```

- **部署**：API 由 **Java 后端** 提供（默认端口 8096）；前端可另起工程，经代理或 CORS 访问 `/api`。  
- **权限**：管理端 API 使用 **JWT + 角色**；`/api/public/**` 多为公开只读。

## 五、与现有数据库设计的衔接与建议扩展

您已有 **Event / Schedule / Record / Registration / Media** 等核心表，可直接支撑 **录入类赛事信息、结果、预告（赛程维度）**。

为支撑 **官网拉取、球员关注、编辑类预告**，建议在逻辑模型上 **增量** 如下（表名可按您命名规范调整）：

| 建议表/实体 | 用途 |
|-------------|------|
| **`cms_article`（或 `med` 扩展）** | 运营撰写的预告、资讯；字段含标题、正文、封面、发布时间、置顶、关联 `event_id` / `athlete_id` |
| **`cfg_featured_athlete`** | 后台指定「重点展示球员」及排序，供首页/预告模块筛选 |
| **`ext_data_source`** | 外部抓取源：名称、基础 URL、解析规则版本、启用状态 |
| **`ext_sync_job` / `ext_sync_log`** | 任务调度记录：开始/结束时间、成功条数、错误信息 |
| **`ext_feed_item`** | 抓取到的外部条目：标题、链接、摘要、发布时间、来源站点、原始 ID；**与本地赛事 ID 可选关联** |

**原则**：本地权威赛程/成绩仍以 **`evt_*` / `sch_*` / `rec_*`** 为准；外部数据单独存区，展示时 **区块化**（例如「外部最新动态」），避免未经审核直接覆盖比分。

## 六、「从官网拉取」专项说明

### 6.1 实施步骤（通用）

1. **确认数据源**：目标页面 URL、列表页结构、是否反爬。  
2. **合规**：阅读对方 ToS；控制频率；用户可见处标注来源。  
3. **解析**：`tt-admin-api` 内 **Java** 服务（`RestTemplate` 拉取 + **Jsoup** 解析 DOM），写入 `ext_feed_item`；`tt.sync.mode=queue` 时经 **RabbitMQ** 异步执行（见 [`02-技术栈.md`](./a-锚定项/02-技术栈.md)）。  
4. **去重**：以 `(source_id, external_url)` 或内容哈希去重。  
5. **展示**：官网只读 API 分页返回；失败时降级为仅展示本地数据。

### 6.2 风险与对策

| 风险 | 对策 |
|------|------|
| 页面改版导致解析失败 | 版本化解析器 + 告警 + 后台「同步失败」可见 |
| 法律与封禁 | 优先 API/RSS；爬虫仅作补充且低频 |
| 数据不一致 | 不自动写入 `rec_result`；仅作资讯流 |

### 6.3 若未来有官方 API

将 **`ext_data_source`** 的 type 设为 `api`，同一套 **`ext_feed_item`** 或映射表写入，可减少前端改动。

| 阶段 | 目标 | 交付物 |

## 九、验收关注点（业务侧）

- 公众端 **只读性能**：列表分页、赛程按日/按赛事筛选。  
- 成绩录入 **权限与审计**（与 `gov_audit_log` 对齐）。  
- 外部数据 **来源可追溯、不误导**（明确「转载自 xxx」）。  
- 球员维度 **数据一致**：同一人多场次聚合正确、时区与状态（未赛/已赛）清晰。

## 十一、切片明细与实现方式

> 下列 **API 路径** 为约定示例，落地时可统一加版本前缀如 `/api/v1`。  
> **错误体** 建议统一：`{ code, message, details? }`；列表统一：`{ list, total, page, pageSize }`。

### S01 — 工程骨架与同域路由

| 项 | 实现方式 |
|----|----------|
| **目标** | 本地可启动；`/api/health` 可用；官网与后台入口路径分离。 |
| **目录** | `tt-shared` + `tt-admin/tt-admin-api`（Spring Boot 3.2.5）；`client/`：Vue 2.7 双入口 + `shared/`，产物 `dist/portal`、`dist/admin`。 |
| **路由** | 后端 **REST**：`GET /api/health`、`/api/public/**`、`/api/admin/**`；前端路由由 Vue Router 管理。 |
| **静态** | 开发：Vite 代理 `/api` → Java；生产：Nginx 托管 `dist` 并反向代理 `/api`。 |
| **实现要点** | 跨域时配置 **CORS**；管理端 **JWT**（Bearer）。 |
| **验收** | `GET /api/health` 返回 200；`POST /api/admin/auth/login` 可取得 token；前端 dev 能打开页面。 |

---

### S02 — 数据库最小表 + 迁移

| 项 | 实现方式 |
|----|----------|
| **目标** | 可重复执行的迁移脚本；最小闭环表齐全。 |
| **最小表集** | `fnd_user`、`fnd_role`、`fnd_user_role`；`evt_event`、`evt_season`（可选先合并进 event 字段）、`evt_venue`、`evt_event_venue`；`sch_stage`、`sch_match`、`sch_match_participant`；`reg_athlete`；`rec_result`。 |
| **工具** | `db/schema.sql`（权威 DDL）+ `data.sql` + Flyway `migration/V*.sql`；**DataInitConfig** 写入管理员（bcrypt）。 |
| **实现要点** | 主键、时间字段、索引：`sch_match(event_id, scheduled_at)`、`sch_match_participant(athlete_id)` 便于球员查询。 |
| **验收** | 空库执行迁移 + seed 成功；ORM/连接池在 S01 服务启动时可连通。 |

---

### S03 — 登录、JWT、后台守卫

| 项 | 实现方式 |
|----|----------|
| **目标** | 仅登录用户可访问 `/api/admin/*`（除 login）；官网 `/api/public/*` 无需登录。 |
| **API** | `POST /api/admin/auth/login`（body: username/password）→ 返回 **JWT** `token`；`POST /api/admin/auth/logout`、`GET /api/admin/auth/me`（需 Bearer）。 |
| **中间件** | Spring Security + `JwtAuthFilter`：校验 `Authorization: Bearer` + `fnd_user_role` 中 `admin` 角色。 |
| **实现要点** | 密码 **BCrypt**；登录限流用 **MySQL** 表 `sys_login_rate`（`LoginRateLimiter`）。 |
| **前端** | 独立前端工程：本地存 token，请求头带 Bearer；路由守卫调 `/api/admin/auth/me`。 |
| **验收** | 未登录访问任意 `POST/PUT/DELETE /api/admin/...` 返回 401；登录后可访问。 |

---

### S04 — 后台：赛事 / 赛季 / 场馆

| 项 | 实现方式 |
|----|----------|
| **目标** | 赛事档案与场馆可维护，为场次挂父级。 |
| **API（admin）** | `CRUD /api/admin/events`、`/api/admin/venues`；赛事与场馆关联 `PUT /api/admin/events/:id/venues`。 |
| **实现要点** | 赛事状态枚举：`draft | published | archived`；列表支持分页与按状态筛选；**审计** 写入 `gov_audit_log`（可与 S09 一并加）。 |
| **前端** | Element 表格 + 表单抽屉；场馆多选关联赛事。 |
| **验收** | 创建赛事 → 在后台列表可见 → S05 可下拉选择该赛事。 |

### S14 — 抓取调度 + 解析器（三期）

| 项 | 实现方式 |
|----|----------|
| **目标** | 定时拉取并写入 `ext_feed_item`，失败写日志。 |
| **库表** | `ext_feed_item`：`source_id, external_url, title, summary, published_at_ext, raw_hash, created_at`；唯一索引 `(source_id, external_url)` 或 `external_id`。 |
| **实现要点** | **策略模式**：`HtmlV1Parser`（`com.tt.application.service.ext`）入参 `(html, baseUrl)` → 条目数组；改版可增 `html_v2` 等 `parser_type`。 |
| **触发与调度** | `POST /api/admin/ext-sources/:id/sync` 手动触发；`ExtSyncScheduler` 按 `interval_minutes` 自动扫描启用源。`tt.sync.mode=direct` 同步执行，`queue` 时投递 **RabbitMQ**（`ExtSyncListener` → `ExtSyncService`）。 |
| **解析器** | `html_v1`（Jsoup 链接抽取）、`rss`（RSS 2.0 XML） |
| **HTTP** | **RestTemplate** + 超时；尊重 robots；频率配置见 `ext_data_source.interval_minutes`。 |
| **验收** | 手动触发后 `ext_feed_item` 有新行且去重生效；失败时 `ext_sync_log` 有 error。 |
